**Answers**
1. DSL: the operator set is not sufficient. Mandatory MVP additions are arithmetic `+ - * /`, unary negation, comparisons for `if_then`, and the `ts_corr/ts_cov` pair already listed in the spec at [spec.md:53](/home/ivena/coding/python/factor-lab/spec.md#L53). `regex` is the wrong tool once nesting and infix operators exist. `pyparsing` is acceptable, but a small hand-written Pratt/recursive-descent parser is not overkill here. `max_length=200` is fine. `max_depth=3` is okay only if you count function nesting, not every binary op; otherwise use 4-5.

2. Walk-forward: with current repo data as of March 18, 2026, US has 444 trading days at [reports/eval_us.md:6](/home/ivena/coding/python/factor-lab/reports/eval_us.md#L6) and CN has 421 at [reports/eval_cn.md:6](/home/ivena/coding/python/factor-lab/reports/eval_cn.md#L6). A 6-month OOS holdout leaves roughly low-300/high-200 gross IS days, and less after 120-day lookback burn-in. So 3 folds are marginal, not impossible. I would default to 2 folds unless effective IS days after burn-in are at least about 300. Use anchored expanding windows for the headline metric; use sliding windows only as a robustness diagnostic. Main look-ahead risks are same-bar close execution, non-point-in-time universe membership, and any normalization/neutralization fit on the full sample instead of inside each fold.

3. Gates: `|IC| > 0.02`, `|IC_IR| > 0.3`, and corr `< 0.7` are reasonable promotion gates, not discovery gates. Monotonicity `> 0.7` is too strict as a hard gate and should be sign-aware. Turnover cannot be trusted until units are fixed. Based on current reports, most factors will not pass `IC > 0.02`; among the baseline set, only one US factor does and no CN factor does. Yes, gates should be market-specific for CN vs US.

4. Agent loop: 50 experiments is enough for an MVP session if you also enforce distinctness and do not OOS-check every near-duplicate. As written, the agent can game PASS/FAIL over repeated sessions. Add formula canonicalization, similarity dedupe, and a rule like “only OOS-check the top 3 distinct IS candidates per day.” Show regime-conditional IC only as coarse IS summaries with sample counts, not highly granular slices. Use the Anthropic SDK directly for the loop; `claude -p` is fine for manual smoke tests, not for a robust 50-step controller.

5. Timeline: 4 days is not realistic for full Phases 1.5 through 3.4. The MVP subset should be: minimal turnover/reporting, DSL parser with arithmetic plus core operators, compute engine, 2-fold expanding walk-forward, 3 core gates, and a manual or semi-automated agent loop. Registry promotion and pipeline injection can come later.

6. Missing considerations: yes on all four. Add simple trading costs/slippage before any promotion decision. Report both equal-weight and a liquidity-aware weighting variant. Treat sector/industry neutralization as an evaluation toggle or post-processing step rather than a first DSL feature. After promotion, monitor rolling 60-day IC, hit-rate, turnover, capacity, and correlation drift, and retire degraded factors automatically.
