**Findings**
1. High: Quintile stats are silently wrong when `qcut` cannot form all 5 bins. [`quintile.py:33`](/home/ivena/coding/python/factor-lab/src/evaluate/quintile.py#L33) uses `duplicates="drop"`, so tied factors can collapse to fewer bins or all `NaN`. The code still iterates `1..5` [`quintile.py:35`](/home/ivena/coding/python/factor-lab/src/evaluate/quintile.py#L35), fills never-seen buckets with `0.0` [`quintile.py:44`](/home/ivena/coding/python/factor-lab/src/evaluate/quintile.py#L44), and reports `Q5-Q1`/`n_days` from those partial series [`quintile.py:52`](/home/ivena/coding/python/factor-lab/src/evaluate/quintile.py#L52). In local checks, an all-constant factor produced `[0,0,0,0,0]` with `n_days=0`, and a two-value factor produced only Q1/Q2 data but still reported a 5-quintile summary.

2. High: `compute_rsi` is not a faithful Wilder RSI implementation. [`factors.py:20`](/home/ivena/coding/python/factor-lab/src/evaluate/factors.py#L20) computes the initial average gain/loss, but the loop at [`factors.py:23`](/home/ivena/coding/python/factor-lab/src/evaluate/factors.py#L23) updates with the next delta before writing any RSI value, so the first valid RSI appears at `period+1` instead of `period`. Also, [`factors.py:26`](/home/ivena/coding/python/factor-lab/src/evaluate/factors.py#L26) maps every `avg_loss ~= 0` case to `100`, which misclassifies flat windows as extreme overbought instead of neutral/undefined. That feeds directly into `rsi_14` and `reversion_score` [`factors.py:129`](/home/ivena/coding/python/factor-lab/src/evaluate/factors.py#L129).

3. Medium: The factor-correlation matrix drops whole dates when any one factor is constant or missing. [`correlation.py:23`](/home/ivena/coding/python/factor-lab/src/evaluate/correlation.py#L23) builds the full daily matrix, and [`correlation.py:25`](/home/ivena/coding/python/factor-lab/src/evaluate/correlation.py#L25) only keeps the day if there are no `NaN`s anywhere. That means a single constant factor removes valid correlations for every other pair on that date, biasing the average toward high-variation days.

4. Medium: The sample is biased against short-history names, and the effective threshold is higher than “60 bars”. [`factors.py:117`](/home/ivena/coding/python/factor-lab/src/evaluate/factors.py#L117) drops symbols with `<60` bars, [`factors.py:141`](/home/ivena/coding/python/factor-lab/src/evaluate/factors.py#L141) only emits factor rows starting at index `60`, and the merge then requires non-null `fwd_5d` [`run_evaluation.py:94`](/home/ivena/coding/python/factor-lab/src/evaluate/run_evaluation.py#L94) [`run_evaluation.py:98`](/home/ivena/coding/python/factor-lab/src/evaluate/run_evaluation.py#L98). In practice a symbol needs about 66 bars to appear once. That is not pure survivorship bias by itself, but it is a history-length selection bias against IPOs, recent listings, and short-lived names.

5. Medium: `ic_ir` is only a heuristic here, not a valid significance measure. The report uses overlapping daily `fwd_5d` returns [`run_evaluation.py:94`](/home/ivena/coding/python/factor-lab/src/evaluate/run_evaluation.py#L94), while `ic_summary` computes `mean / std` as if daily ICs were independent [`ic.py:37`](/home/ivena/coding/python/factor-lab/src/evaluate/ic.py#L37). Overlap induces serial correlation, so the reported IR is directionally useful but statistically optimistic.

6. Low: SQL identifiers are interpolated directly into queries. [`forward_returns.py:22`](/home/ivena/coding/python/factor-lab/src/evaluate/forward_returns.py#L22) and the raw-price query in [`run_evaluation.py:82`](/home/ivena/coding/python/factor-lab/src/evaluate/run_evaluation.py#L82) both use f-strings for table/column names. With the current hardcoded configs this is not an immediate exploit, but the function API is unsafe if reused with user-controlled identifiers.

**Answers To Your Checks**
- IC: Spearman is the standard cross-sectional IC measure here. The `groupby(date)` daily IC approach in [`ic.py:21`](/home/ivena/coding/python/factor-lab/src/evaluate/ic.py#L21) is also standard; the main tradeoff is that it equal-weights dates, not stock observations.
- Reversion score sign: the sign convention is logically correct in [`factors.py:131`](/home/ivena/coding/python/factor-lab/src/evaluate/factors.py#L131) to [`factors.py:134`](/home/ivena/coding/python/factor-lab/src/evaluate/factors.py#L134). Positive means oversold / expect up. The caveat is the RSI bug above, which can make neutral flat windows look overbought.
- Forward returns: `LEAD(close, h) / close - 1` in [`forward_returns.py:18`](/home/ivena/coding/python/factor-lab/src/evaluate/forward_returns.py#L18) is the correct forward `h`-trading-day return and does not introduce look-ahead bias, assuming one row per symbol/date. The only modeling caveat is execution timing: close-based factors paired with close-to-close returns assume you can act after seeing the close.
- Correlation matrix: averaging daily cross-sectional rank correlations is the right concept for redundancy analysis; the implementation just needs pairwise-valid-day handling instead of all-or-nothing date dropping.
