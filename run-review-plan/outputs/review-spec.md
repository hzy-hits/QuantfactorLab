**Findings**

1. The spec does not model US and CN as separate deployment domains, so factor identity, gating, and lifecycle are conflated across incompatible markets. `[spec.md:78](/home/ivena/coding/python/factor-lab/spec.md#L78)` `[spec.md:139](/home/ivena/coding/python/factor-lab/spec.md#L139)` `[spec.md:216](/home/ivena/coding/python/factor-lab/spec.md#L216)` `[spec.md:238](/home/ivena/coding/python/factor-lab/spec.md#L238)` `[spec.md:272](/home/ivena/coding/python/factor-lab/spec.md#L272)`  
   `factor_id = hash(formula)` and a single `status` row are not enough when the same formula can be valid in CN, invalid in US, or depend on CN-only fields like `margin_balance` / `net_mf_amount`. The same issue applies to shared gates like `IC > 0.02` and `turnover < 40%/month`: those may not mean the same thing in both markets. This needs a split between market-agnostic factor definitions and market-specific factor instances/deployments.

2. The write-back design creates both a dependency loop and an unsafe DuckDB concurrency boundary. `[spec.md:270](/home/ivena/coding/python/factor-lab/spec.md#L270)` `[spec.md:274](/home/ivena/coding/python/factor-lab/spec.md#L274)` `[spec.md:295](/home/ivena/coding/python/factor-lab/spec.md#L295)`  
   Factor Lab is described as reading pipeline DBs as upstream inputs, then promotion writes back into those same DBs. If Factor Lab ever reads tables influenced by `lab_factor`, you have a self-referential research loop. Separately, DuckDB officially supports one read/write process or many read-only processes, but not automatic multi-process writes; cross-process writers need app-level locking/retry. It also does not support writing to multiple attached databases in one transaction. So direct writes into both US and CN pipeline DBs are not a safe default design. A cleaner boundary is: Factor Lab writes only to its own DB or Parquet export, then each downstream pipeline imports from that artifact in its own controlled writer job.

3. Raw `IC_IR` weighting is too weak as the composite-construction rule. `[spec.md:127](/home/ivena/coding/python/factor-lab/spec.md#L127)` `[spec.md:143](/home/ivena/coding/python/factor-lab/spec.md#L143)` `[spec.md:272](/home/ivena/coding/python/factor-lab/spec.md#L272)`  
   `IC_IR` is a reasonable admission signal, but as a weight it ignores pairwise correlation, turnover/cost, sign instability, and market-specific covariance. In practice it will over-allocate to redundant factors that all passed the same gate. For v1, I would trust either equal-weight after correlation clustering, or shrunk correlation-adjusted weights, more than raw `IC_IR` weights. Keep separate composites per market.

4. The auto-retirement rule is brittle as a hard retirement trigger. `[spec.md:126](/home/ivena/coding/python/factor-lab/spec.md#L126)` `[spec.md:278](/home/ivena/coding/python/factor-lab/spec.md#L278)`  
   A 60-day rolling IC on 5D forward returns is noisy, and “20 consecutive days” is based on heavily overlapping windows, so it is not 20 independent confirmations. I would treat this as a watchlist or down-weight signal, not immediate retirement. Hard retirement should use a longer live window, confidence-adjusted live IR/IC, or a two-stage state transition like `promoted -> watchlist -> retired`.

5. The DuckDB schema is under-specified for auditability and market separation; extra secondary indexes are probably not the main fix. `[spec.md:215](/home/ivena/coding/python/factor-lab/spec.md#L215)` `[spec.md:244](/home/ivena/coding/python/factor-lab/spec.md#L244)` `[spec.md:252](/home/ivena/coding/python/factor-lab/spec.md#L252)`  
   Missing columns include `market`, `universe`, `security_id` or canonical symbol mapping, `fwd_horizon`, `rebalance_freq`, `neutralization`, `data_snapshot_id`, `computed_at`, and deployment-specific weight/state. `ts_code` is CN-centric, and `oos_pass BOOLEAN` is not enough for audit; hidden OOS metrics should still be stored in a restricted table. `max_corr_factor/max_corr_value` are also too lossy; use a separate evaluation/correlation table. On indexes: DuckDB already creates ART indexes for `PRIMARY KEY`/`UNIQUE`, and official guidance is to avoid explicit extra indexes unless queries are highly selective. The bigger issue here is missing keys/columns and a better market-scoped schema, not more indexes.

**Bottom Line**

US/CN handling is not clean yet. The spec needs a `MarketConfig`/adapter layer plus market-scoped factor deployments, separate composites, and separate promotion state.

`IC_IR` weighting is acceptable as a bootstrap baseline, not as the long-term composite design. The retirement rule is too brittle for hard deletion. The main architectural risk is the write-back path into pipeline DBs: safe only if you serialize writers, keep Factor Lab from reading its own outputs, and run promotions as separate per-market jobs.
