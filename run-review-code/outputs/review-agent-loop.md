**Findings**
1. The IS/OOS split leaks post-cutoff returns into IS scoring. `_compute_forward_returns()` attaches a 5-day `LEAD()` return to the start date, but `walk_forward_backtest()` keeps all rows with `date < oos_start`, so late-September rows still use October prices. A factor dated `2025-09-30` is still scored as IS even though its `fwd_5d` extends past `2025-10-01`. See [loop.py:99](/home/ivena/coding/python/factor-lab/src/agent/loop.py#L99) and [walk_forward.py:263](/home/ivena/coding/python/factor-lab/src/backtest/walk_forward.py#L263).

2. OOS testing ignores the gate results. The gate module says all gates must pass before OOS, but the session picks the top 3 from every valid factor frame, not from `gates_passed` candidates, and it sorts by `abs(is_ic)` rather than the signed `is_ic`. That lets high-turnover, non-monotonic, or redundant factors consume OOS budget and appear in the final report. See [gates.py:4](/home/ivena/coding/python/factor-lab/src/backtest/gates.py#L4), [loop.py:638](/home/ivena/coding/python/factor-lab/src/agent/loop.py#L638), and [loop.py:677](/home/ivena/coding/python/factor-lab/src/agent/loop.py#L677).

3. The prompt advertises a larger DSL and feature set than the loop can execute. The agent is told to use `quantile`, `neutralize`, `vwap`, `amount`, `rsi_14`, fundamentals, etc., but `_compute_basic_features()` does not provide those columns and `_eval_function()` does not implement `quantile` or `neutralize`. There is already a richer compute engine in `src/dsl/compute.py`, but the session bypasses it. This will waste experiments on formulas the parser accepts but the evaluator rejects. See [prompts.py:18](/home/ivena/coding/python/factor-lab/src/agent/prompts.py#L18), [loop.py:119](/home/ivena/coding/python/factor-lab/src/agent/loop.py#L119), [loop.py:258](/home/ivena/coding/python/factor-lab/src/agent/loop.py#L258), [loop.py:349](/home/ivena/coding/python/factor-lab/src/agent/loop.py#L349), and [compute.py:37](/home/ivena/coding/python/factor-lab/src/dsl/compute.py#L37).

4. The “avoid correlated existing factors” rule is not actually enforced. The prompt promises auto-reject above `corr > 0.7`, but the session passes `existing_factors=None` into the prompt builder and never passes candidate/existing series into `check_gates()`, so the correlation gate always auto-passes. See [prompts.py:97](/home/ivena/coding/python/factor-lab/src/agent/prompts.py#L97), [prompts.py:113](/home/ivena/coding/python/factor-lab/src/agent/prompts.py#L113), [loop.py:549](/home/ivena/coding/python/factor-lab/src/agent/loop.py#L549), [loop.py:638](/home/ivena/coding/python/factor-lab/src/agent/loop.py#L638), and [gates.py:135](/home/ivena/coding/python/factor-lab/src/backtest/gates.py#L135).

5. `_ask_agent()` works syntactically, but the SDK and CLI paths are not equivalent. The “system prompt” is not sent via Anthropic’s `system=` field; it is concatenated into the user prompt. The CLI fallback also ignores `self.model`, so fallback may silently use a different model than the SDK path. `resp.content[0].text` is also brittle if the first block is not text. See [loop.py:563](/home/ivena/coding/python/factor-lab/src/agent/loop.py#L563), [loop.py:727](/home/ivena/coding/python/factor-lab/src/agent/loop.py#L727), and [loop.py:740](/home/ivena/coding/python/factor-lab/src/agent/loop.py#L740).

6. Response parsing and failure recovery are weak enough to waste budget. The regex parser only really handles the exact label form; on a realistic variant like `**FORMULA:** ...`, it mis-parses the later fields into `formula` instead of rejecting cleanly. Then DSL parse failures and compute crashes are both flattened into a generic `compute_error`, and the next prompt only shows zeros rather than the real error reason, so the agent cannot self-correct. See [prompts.py:196](/home/ivena/coding/python/factor-lab/src/agent/prompts.py#L196), [prompts.py:243](/home/ivena/coding/python/factor-lab/src/agent/prompts.py#L243), [prompts.py:169](/home/ivena/coding/python/factor-lab/src/agent/prompts.py#L169), [loop.py:574](/home/ivena/coding/python/factor-lab/src/agent/loop.py#L574), and [loop.py:602](/home/ivena/coding/python/factor-lab/src/agent/loop.py#L602).

**Answers**
1. `prompts.py`: structurally decent, but not reliable because it gives false context. The biggest problem is mismatch with the actual evaluator and missing live context (`regime_dist=None`, `existing_factors=None`).
2. `parse_agent_response`: not robust enough. If the agent drifts from the exact format, it can return `None` or, worse, mis-parse garbage as a formula.
3. `loop.py` flow: the high-level order is correct, but OOS selection is wrong because it ignores gates, and the IS/OOS split leaks across the `2025-10-01` boundary.
4. `_ask_agent`: fallback is partial, not correct parity. SDK path and CLI path do not use the same model/prompt semantics.
5. OOS leakage: direct “only top 3 at end” leakage is mostly prevented, but the forward-return boundary leak contaminates IS metrics.
6. Error handling: response-parse failure becomes `parse_error`; DSL parse failure and compute crash both become generic `compute_error`; API failure logs and burns the iteration without recording structured feedback.
7. Infinite loop / spend: no infinite loop risk, because the loop is bounded by `budget`. Spend is still weakly controlled because failures still consume iterations, there is no early abort/backoff, and prompt size grows with full experiment history on every call.

No automated tests cover these paths in this repo, so this is a static review.
